# 房态与订单运营流程分步开发计划

## 0. 执行方式

用户审阅、启动与逐步验收请统一从[《QinTopia PMS 分步开发与人工验收计划》](./QinTopia-PMS-分步开发与人工验收计划.md)进入；本文件不作为要求用户逐条阅读的入口。

本计划记录用户在现有 RoomStatusGrid 上确认和仍待核定的新一轮运营工作流改造。当前仅供完整讨论与审阅，不构成实施授权；只有用户明确说“开始实施第 1 步”后，才按人工入口中的 6 个步骤执行。本文件的 15 个技术阶段是步骤内的小检查点：每个技术阶段仍单独交付和验收，但只有同一人工步骤内的全部阶段通过后，才把人工入口推进到下一步。

每个阶段均按以下闭环执行：

1. 实施该阶段的 Web、API、领域、数据库及测试改动，不只修改 UI。
2. 运行该阶段的聚焦自动化测试，并重新启动可试用实例。
3. 向用户提供实际访问 URL、测试结果和逐步点击说明。
4. 停止进入下一阶段，等待用户回复“通过”或提出修改意见。
5. 若用户不通过，只修复当前阶段并重新交付；不得提前夹带后续阶段。
6. 用户通过后固定当前行为，再开始下一阶段。

每次人工验收使用独立、可识别且无冲突的订单或日期夹具。取消、未到、退房等互斥终态必须使用不同订单，不能让前一项操作污染后一项结果。交付验收前还要确认浏览器控制台无新增错误、关键网络请求状态正确，并说明是否保留了本次验收数据。

一个阶段只有同时满足以下条件才算“可提交验收”：

- 页面使用真实 API 和 PostgreSQL，不使用 mock、占位数据或静态演示结果；
- Web 与 API 仍使用相同领域规则、授权、事务、幂等和持久结果；
- 聚焦的 unit/domain、integration、contract 和 browser E2E 已通过；
- 可试用实例的 health/readiness 正常；
- 工作人员可按本文件的人工验收步骤独立完成操作；
- 不在工作人员主流程中暴露实现协议或内部代码。

## 1. 文档权威与覆盖关系

1. `QinTopia-PMS-分步开发与人工验收计划.md` 是用户逐步审阅、启动和验收的唯一入口；本文件是它的工程实施清单。两者冲突时，先暂停对应切片并修正文档，不允许开发人员自行选择解释。
2. `spec-qintopia-pms-core-operations-mvp.md` 继续负责库存、计价、会员、资金、事务和审计不变量；本计划只在明确列出的用户最新决定上精确覆盖旧表述，不把展示偏好扩展成其他领域变化。核心规格中尚未同步的旧会员有效期、退费和参考产品表述，应在实施对应阶段时一并修订，不能继续作为运行规则。
3. 本计划负责最新确认的工作人员操作流程、字段渐进显示、中文文案和逐阶段人工验收顺序。
4. `房态90天连续时间轴与超长住宿开发规格.md` 是本计划最后一个主要功能阶段的详细规格；本计划不把其 90 天验收压缩或替代。
5. `design-system/qintopia-pms/MASTER.md` 继续负责视觉、响应式和无障碍基线。
6. 所有 companion（包括 90 天规格和旧 RoomStatusGrid 目标）中的 Preview、Confirm、Receipt、Quote ID、policy ID 等协议仍是后端/API 合同，但其工作人员可见文案和交互步骤全部被本计划覆盖：日常操作页面不得直接展示这些名称和内部 ID。
7. 外围智能体 API、审计查询和故障恢复仍保留稳定 Preview、Confirm、Receipt、command、correlation 和幂等语义；隐藏技术词不等于删除协议。
8. 核心规格中“所有新 Order 必须选择四个渠道之一”的旧表述，自本计划起收窄为“所有新建非会员正常住宿订单”。免费入住和会员住宿是明确例外，渠道和渠道订单号都必须保存为 `null`；会员权益不足而产生未覆盖金额也不改变住宿来源。绕过 Web 提交会员渠道必须由服务端与数据库拒绝且零业务写入。这不改变历史空值原样兼容规则，也不新增第五个渠道。

## 2. 已确认的人员与业务身份

必须始终区分三个主体：

- **登录工作人员：** 当前登录 PMS、执行操作和留下审计记录的人。
- **主要居住人：** 实际入住的住客，保存为订单成交时的不可变快照。
- **权益所属会员：** 会员权益的所有人；只有勾选使用会员权益时才选择，可能与主要居住人相同，也可能不同。

系统目前不是会员自助系统。页面不得把登录工作人员描述成会员本人，也不得要求工作人员选择或理解会员合同 ID。

## 3. 业务类型与字段边界

| 业务类型 | 是否有入住人 | 入住人信息 | 渠道 | 金额与会员 | 最终事实 |
|---|---:|---|---|---|---|
| 非会员正常住宿 | 是 | 姓名、昵称必填；联系电话、证件号沿用现有住客字段与校验 | 必选游牧岛、携程、美团或企业微信；企业微信渠道订单号必须为空 | 自动计价；可人工指定最终总价 | Order + Stay + Claim |
| 会员住宿 | 是 | 与正常住宿复用同一个住客组件、同一组字段和同一套校验 | `bookingChannelCode=null`，`channelOrderReference=null` | 权益可全额、部分或零余额覆盖；未覆盖部分按规则计价 | Order + Stay + Claim + Entitlement Coverage |
| 免费入住 | 是 | 与正常住宿复用同一个住客组件、同一组字段和同一套校验 | `bookingChannelCode=null`，`channelOrderReference=null` | 金额始终为 0；不得使用会员权益 | Order + Stay + Claim |
| 内部占用（边界待核定） | 待确认 | 候选方案为不显示、不保存入住人 | 不适用 | 候选方案为不计价、不收款、不使用会员权益 | 候选方案为 INTERNAL_USE Block + Claim |
| 维修锁房 | 否 | 不显示、不保存入住人 | 不适用 | 不计价、不收款、不使用会员权益 | MAINTENANCE Block + Claim |

免费入住还必须填写：

- 免费入住类型 `freeStayCategoryCode`：`VOLUNTEER`（义工）或 `RECEPTION`（接待）；
- 本次免费入住的具体原因；
- 房源、入住日期和退房日期。

数据库列 `free_stay_category_code` 对历史免费订单允许为 `null`，Query/DTO 原样返回，Web 只派生显示“历史未记录”；所有新建免费入住 Command 必须提供两个合法 code 之一。迁移不得把未知历史行虚构回填成义工或接待。暂不增加“其他”免费类型，不虚构批准人或审批流。免费入住的入住、续住、缩短、换房、取消和退房与普通住宿使用相同履约模型；每次变更追加 amendment、pricing revision 和审计，金额保持 0。

## 4. 工作人员可见语言

主流程使用以下中文业务语言：

- `正在计算住宿金额`
- `住宿金额`
- `计价依据`
- `本次住宿使用会员权益`
- `会员权益覆盖晚数`
- `未覆盖晚数`
- `权益未覆盖金额`
- `订单概述`
- `确认创建订单`
- `订单已创建`
- `房态或价格已经变化，请重新核对`
- `正在查询刚才的操作结果`

下列内容不得直接出现在工作人员的报价、订单概述、成功页或常规订单详情主区域：

- `Preview`、`Confirm`、`Receipt`、`Command`；
- `Quote ID`、`Receipt ID`、`Command ID`、correlation ID、幂等键；
- pricing policy code、policy version ID、Lot ID、Coverage ID；
- `coverageSet`、`cashRemainder`、`currentContractAmount` 等字段名；
- “获取服务端报价”“现金计价行”“资源引用”等开发或协议语言。

内部稳定引用继续保存在 API、数据库、审计和故障恢复中。获权的技术审计入口可以单独查看，但不能成为完成日常订单的必经页面。

### 4.1 房态状态与机器事实

- 工作人员房态统一使用“已预订”和“在住”：订单创建且尚未办理入住时显示“已预订”；办理入住成功后立即显示“在住”。“已入住”只描述一次已经发生的动作，不作为持续状态；“入住中”不再作为另一套同义状态。
- 办理退房后，订单详情显示“已退房”；房态不再把该订单表达为当前在住。退房日如存在待办清洁则另行显示“待清洁”，完成后显示清洁任务“已完成”。“是否可售”继续由库存事实和待确认的清洁经营规则共同决定，不能先假设清洁状态必然阻断或放开销售。历史日期仍可追溯当日住宿事实，但不得让工作人员误解为住客当前仍在住。
- 正常订单或免费入住已经占用库存时，服务端仍必须保留 `blocking`、`conflicts`、Claim 和完整冲突引用，用于可售判断、并发互斥和确认时重新校验；工作人员格子、Tooltip、屏读名称及常规详情不显示“阻断”“阻断库存”“N 个阻断”“库存 Claim”等机器语言。
- “已预订”或“在住”本身已经说明该日期不能重复出售，不再叠加红色阻断数量。只有出现真正需要工作人员处理的异常，或工作人员尝试在已占日期创建业务时，才用中文业务句子说明具体对象和日期，例如“101A 在 7 月 23 日至 28 日已有住宿，不能重复安排”，而不是暴露冲突对象代码。

## 5. 目标主流程

工作人员完成一笔正常订单时，应只有以下可感知步骤：

1. 在房态网格拖选房源和日期，或在右侧输入房源、入住日期和退房日期。
2. 系统自动检查完整区间，并显示总晚数、住宿金额和简明计价依据。
3. 工作人员填写主要居住人、渠道等当前订单需要的信息；只有勾选会员权益后才出现会员搜索。
4. 点击“创建订单”，进入不写业务事实的订单概述。
5. 核对住客、房源、日期、晚数、会员覆盖、标准金额和最终金额。
6. 如需改价，在概述中勾选“调整最终总价”，填写非负整数元最终总价和必填原因。
7. 点击一次“确认创建订单”。服务端重新校验并原子创建订单。
8. 成功页只显示订单号、住客、房源、日期、金额和下一步可执行操作。

后端仍在上述流程中执行高风险预检、显式确认、重新校验、幂等和持久结果，但工作人员不需要理解协议名称或复制任何代码。

### 5.1 操作意图、变更原因与调价原因

三种“原因”不得混为一个自由文本字段：

- **创建普通订单的操作意图：** 工作人员无需填写原因。Web/API 仍在命令协议中保存与命令类型绑定的稳定业务原因码，例如 `CREATE_STANDARD_ORDER`，`note` 为空；该码只说明正在创建普通订单，不得伪造成工作人员陈述。登录主体、时间和 correlation 继续由审计事实记录。
- **住宿变更原因：** 续住、缩短、换房、取消、未到以及免费住宿变更按各阶段要求由工作人员填写，保存为该 amendment 的业务原因。
- **调价原因：** 只有工作人员勾选“调整最终总价”时才显示并必填，专门解释为什么偏离政策基础报价，不能用命令原因码代替。

外围智能体仍必须按 API 合同提交获准的稳定 reason code；Token 不得因此绕过 reason、Preview、显式 Confirm 或审计。服务端必须验证 reason code 与 command type 匹配，不能接受任意客户端文本冒充结构化操作意图。

付费订单的创建、续住、缩短和换房概述都使用同一调价规则：默认采用政策重算金额；只有本次明确勾选调价时才接收新的非负人民币整数元最终总价，并展示政策基础价、目标价和差额。本次调价只属于本次 revision，任何后续 revision 都不继承。

## 6. 当前已知缺口

实施前以当前工作树复核，已知至少包括：

1. Web 将新付费订单固定为 `TRANSIENT`，导致 7 夜以上被服务端正确拒绝；104 房间 `2026-07-26` 至 `2026-08-05` 的 10 夜订单目前无法报价。
2. 前端把部分确定性 4xx 校验失败误判为网络结果未知，错误进入恢复锁定状态。
3. 住宿类型仍由工作人员选择，且存在“获取服务端报价”按钮。
4. 房态侧栏存在“应用选区”和“清除选区”两个多余中间动作。
5. 报价区域展示逐日英文现金行、内部政策代码、Quote/Receipt ID 和字段名。
6. 创建订单存在多层 Preview/Confirm/Receipt 技术页面，工作人员难以判断真正需要确认几次。
7. `/api/v1/orders/:id` 的 amendment payload 序列化与 schema 不一致，真实订单详情可能返回 500，间接隐藏续住、缩短、换房、入住和退房入口。
8. 会员搜索与会员合同默认暴露，普通订单也显示 `coverageSet 0 晚`，容易被误解为住宿晚数为 0。
9. 多人间父房格虽然已取得完整逐床住客列表，但当前只显示“首个昵称 +N”，其余昵称只有悬停后可见，不符合首页直接显示每位住宿人的最新要求。
10. 90 夜查询上限已存在，但连续 90 天时间轴、约 14 天可见窗口和拖选边缘自动滚动尚未完成。
11. 点击已占日期当前只形成单日选区，无法按稳定订单/Stay 引用选中完整连续住宿；右侧仍停留在日期草稿与报价上下文，不能直接查看该订单的业务概述和当前可执行操作。
12. 正常已预订订单的每个日期格仍额外显示“1 个阻断”，Tooltip、屏读名称和右侧详情还暴露“阻断库存、阻断冲突、库存 Claim”等机器协议；该信息只是库存不可重复出售的内部依据，不是工作人员需要处理的异常。
13. 会员当前只有零散档案/合同能力，缺少工作人员可用的统一会员列表、会员订单、多笔企微收款、生效动作、权益余额与不可变交易/核销记录闭环。
14. 当前 catalog 仍把会员产品标为参考数据、把有效期起点写成付款日、把退费写成不可退款；最新确认规则是工作人员点击“生效”后起算一年，退费在合同拟定前不进入当前版本，不能继续沿用旧运行语义。
15. D/E 楼仍显示 `D-GEN-*`、`E-GEN-*`；应改为 `D01-D05`、`E01-E03`，继续保留 `PMS_GENERATED` provenance 和内部不可变库存 ID。
16. 资金表单目前按最小货币单位录入，而人工调价按元录入，存在 100 倍误录风险；工作人员端应统一按人民币元输入和显示，API/数据库继续使用精确 minor units。
17. 今日履约页面遗漏“到店日已过、仍为已预订且尚未超过离店日”的订单；本地恢复记录损坏时会阻断当前浏览器会话的写操作，但没有工作人员可执行的安全恢复路径。这两项是发布前门禁，不阻断阶段 1 的报价改造。
18. 房态双栏当前按浏览器 viewport 过早启用，没有扣除 `210px` 左导航和页面内边距；常见笔记本上右侧上下文会把房态主表挤成狭小窗口。页面也缺少针对短屏的紧凑密度和典型 `1280x720`、`1366x768`、`1440x800` 几何回归。该缺口归入人工第 3 步的阶段 7 修复，不提前夹带到正在执行的第 2 步。

## 7. 分阶段实施与人工验收

### 阶段 1：选房后自动显示住宿金额

**目标：** 工作人员选择房源和日期后，不再选择住宿类型、不再应用选区、不再主动请求报价，系统直接显示权威住宿金额。

**实施范围：**

- 网格拖选结束后立即同步右侧房源和半开日期；侧栏修改房源或日期后立即同步当前网格可见交集。
- 桌面横向拖选锁定起始房间或床位行，只按鼠标横坐标更新日期；纵向漂移到相邻行不得切换行、停止日期更新或显示相邻行 hover，重复落在同一日期不得触发重复刷新。
- 删除主流程中的“应用选区”和“清除选区”。新选区直接替换旧草稿；关闭办理区域只放弃未提交草稿，不写业务事实。
- 删除付费住宿的“临住、周住、月住、自定义周期、固定期限、滚动续期”等人工选择。

### 阶段 2：正常订单概述与一次确认

**目标：** 用工作人员能理解的订单概述代替技术 Preview/Confirm/Receipt 页面。

**实施范围：**

- 正常订单表单只显示本阶段需要的主要居住人、房源日期、渠道和金额。
- 主要居住人姓名、昵称必填；联系电话和证件号沿用现有字段与领域校验。
- 渠道持久化稳定 code，界面显示中文；WECOM 隐藏或禁用渠道订单号并强制保存 `null`。
- “创建订单”只打开订单概述，不产生 Order/Stay/Claim。
- 概述页后台取得与当前表单完全绑定的服务端预检结果，显示住客、房源、日期、晚数、渠道、标准金额和最终金额。
- 默认不改价，不显示原因字段。勾选“调整最终总价”后才显示目标最终总价、差额和必填原因。
- 正常创建不要求工作人员填写自由文本原因；命令保存与 `CREATE_ORDER` 绑定的稳定操作意图 reason code 和空 note，不能伪造工作人员说明。
- 人工目标价与初始订单在同一原子命令中形成首个完整 pricing revision，不能先建原价订单再二次修改。
- 概述只提供“返回修改”和“确认创建订单”；正式业务确认只有一次。
- 成功页不再要求查看 Receipt 或资源引用，只显示业务结果和下一步。
- 在本阶段同时完成 amendment payload 查询 DTO/schema 的最小一致性修复，确保成功页的“查看订单”能够打开刚创建的真实订单；完整详情布局和动作可发现性仍在阶段 7 收口。

**自动化门禁：**

- 四个合法渠道及 WECOM 强制空渠道订单号通过 domain/integration/contract/E2E。
- 未填写姓名、昵称或渠道时零业务写入。
- 默认价与人工指定最终价分别产生正确且不可变的首个 revision。
- 人工调价原因只在改价时必填；最终价必须为非负人民币整数元。
- 服务端只接受与命令类型匹配的稳定操作意图 reason code；普通创建的空 note 与人工调价的必填原因分别通过 contract test。
- 重复确认或网络恢复不重复产生订单、Stay、Claim 或 revision。
- 新创建的订单详情 Query 不再因 amendment payload schema 返回 500。

**用户人工验收：**

1. 使用阶段 1 的 104 房间 10 夜草稿填写住客姓名、昵称和渠道。
2. 点击“创建订单”，确认只进入一页中文“订单概述”。
3. 不改价直接确认，检查成功页是否只出现业务信息和下一步操作。
4. 从成功页打开刚创建的订单，确认可以正常进入详情而不是内部错误。
5. 再建一笔订单并勾选“调整最终总价”，确认只有此时才要求填写原因。
6. 故意返回修改日期，确认旧概述不会被直接提交，必须重新核对更新后的金额。

**停点：** 用户确认正常订单创建流程和文案通过后，才能进入阶段 3。

### 阶段 3：免费入住

**目标：** 免费入住是一笔有真实住客、正常履约但金额为 0 的住宿订单。

**实施范围：**

- “创建免费入住”进入独立业务模式，但复用正常住宿同一个主要居住人表单组件、字段、校验和不可变快照。
- 必选“义工”或“接待”，并填写具体免费原因。
- 不显示会员复选框、会员搜索、订单来源和渠道订单号。
- 数据库、Command、Preview effect、Confirm、Receipt/Query 和 OpenAPI 保存 nullable 渠道以及稳定免费类型和原因。
- currentContractAmount 始终为 0，不创建会员 coverage，不产生电费或其他现金计价项。
- 免费订单概述和成功页使用与正常订单一致的人类语言，只把金额明确显示为“免费”。

**自动化门禁：**

- 缺失姓名、昵称、免费类型或原因时拒绝且零写入。
- 免费订单的渠道、渠道订单号和会员合同均为 `null`。
- 重放不重复创建订单；房床互斥与普通订单完全相同。
- Query/详情能追溯入住人、免费类型和原因。
- migration、Command schema、effect、Confirm、Receipt、Query DTO、OpenAPI 和 Web 对 `freeStayCategoryCode` 使用同一枚举；历史 `null` 不被伪造回填。

**用户人工验收：**

1. 在空闲房源选择日期并点击“创建免费入住”。
2. 确认需要填写与普通住客相同的姓名、昵称等入住人信息。
3. 选择“义工”或“接待”并填写具体原因。
4. 确认页面不出现渠道、会员和收费字段，订单概述显示金额为免费。
5. 创建后在房态上看到有住客昵称的免费住宿，而不是内部占用。

**停点：** 用户确认免费入住资料和房态表达通过后，才能进入阶段 4。

### 阶段 4：会员档案、会员订单与会员入住

**目标：** 工作人员能从统一会员列表完成建档、出售会员产品、多笔企微收款、生效、余额查看和会员住宿；余额来自不可变 Lot/Ledger，不是可以直接覆盖的档案字段。

**实施范围：**

- 建立统一会员列表和详情。会员档案支持手工录入姓名、身份证号、手机号和微信号；身份证号是唯一业务键，内部关联始终使用不可变 `memberId`，手机号和微信号不承担唯一性。
- 支持按姓名、身份证号、手机号和微信号搜索。会员详情显示当前 ROOM_NIGHT/BED_NIGHT 余额、会员订单、权益 Lot/Ledger、冻结、核销、释放和显式调整记录。
- 飞书入住申请只作为外部申请引用：以 `provider=FEISHU_BASE + base/table + external_record_id` 唯一标识一次申请并关联 `memberId`；同一人多个 record_id 可以关联同一会员。Base 不能直接发放、冻结或核销权益，PMS 正确性不能依赖回写成功，也不得复制现有 workflow 的认证令牌。
- 会员间夜只能由已生效会员订单或显式权益调整事实产生。若工作人员需要更正余额，Web 以“更正后剩余间夜数 + 原因”为业务输入，服务端计算调整量并追加 Ledger 事实；不得覆盖档案余额或删除旧核销记录。
- 将源表中的三个 `REFERENCE_ONLY` 会员产品固化为版本化、可执行且可种子的会员产品 catalog，不建设通用会员产品编辑器：

| 产品 | 默认价 | 发放 | 允许房型/销售单位 |
|---|---:|---:|---|
| 公卫单人间会员 | `¥1,620` | 30 ROOM_NIGHT | `shared_bath_single` 独立房间：`D01`、`D02`、201、205、301-304 |
| 独卫单人间会员 | `¥2,160` | 30 ROOM_NIGHT | `private_bath_single` 独立房间：B01、B02、C01-C04、E02 |
| 公卫四人间会员 | `¥936` | 30 BED_NIGHT | `shared_bath_quad` 单床：101、102、103、105、107、108、109、202、203、206 的 A/B/C/D；不含整房组合 |

- 会员订单保存稳定 `membershipOrderId`、产品、标价、成交价、调价差额、调价原因、状态、生效时间和完整审计。Web/API 输入的是工作人员指定的成交价；未偏离默认价时不要求原因，偏离时原因必填。成交价必须是非负人民币整数元。
- 一张会员订单允许多笔独立企微收款，每笔保存自身外部交易单号。会员订单上下文本身已经限定这是企微收款，不向工作人员提供其他支付方式选项，也不得因此新增通用 `transactionChannelCode`；这条事实只适用于会员订单，不得推断普通住宿资金的 `method`。
- 会员订单在工作人员点击“生效”前至少要有一笔有效企微收款；收款合计与成交价不一致时只显示差额，不阻断显式生效。有效期从服务端记录的生效时点起算一年，并原子发放 30 个对应类型权益。
- 会员订单收款的交易号或金额录错时，提供“更正”命令：冲销原资金事实并追加正确事实；权益间夜更正同样追加调整事实。工作人员可看到当前有效结果和完整历史，但任何已存在的资金、核销或审计事实不可原地编辑或删除。
- 当前版本不实现会员合同退费，也不创建退费公式、退款命令或“已用权益按日报价结算”逻辑。由于会员合同尚未拟定，Web/规格只表达“当前版本不处理会员退费”，不得将其扩大为法律意义的不可退款承诺；旧 `NON_REFUNDABLE_MEMBERSHIP` 运行表述必须退役并保留被最新决定覆盖的 provenance。
- 普通住宿默认不显示会员搜索、会员合同和覆盖区域；提供未默认勾选的“本次住宿使用会员权益”。勾选后显示统一会员选择器，工作人员选择会员档案而不是合同 ID。
- 勾选会员权益后，订单来源渠道和渠道订单号立即隐藏并清空；全额、部分和零余额覆盖都按会员住宿处理，CREATE_ORDER 不提交渠道。非会员普通住宿仍必须选择四个合法渠道之一。
- 会员住宿的主按钮为“核对并创建订单”；核对、成功、持久化恢复和订单详情只显示中文业务语义。订单详情显示“住宿来源：会员权益”，不得显示渠道字段、Preview、Receipt、Command、幂等键或内部 ID。
- 选择会员后，以会员姓名同时预填主要居住人昵称和姓名，以会员手机号、身份证号预填联系电话和证件号码；微信号不映射为入住人昵称。这四项是可编辑的本次住宿快照，报价刷新不得覆盖人工修改，CREATE_ORDER 必须保存提交时的快照值。
- 服务端按 `memberId` 查找日期有效、余额可用且 ROOM_NIGHT/BED_NIGHT 与库存单位匹配的 Lot/Ledger，按具体住宿日期形成 coverageSet 并冻结；同一日期不得重复覆盖，并发请求不得超额冻结。
- 权益不足不拒绝订单。先覆盖可用日期，剩余日期统一按锁定政策版本的 P1 逐晚计算现金余量，最终总价只舍入一次；后续新增有效会员订单可以覆盖尚未覆盖日期并重新计算现金余量。
- UI 只显示总住宿晚数、覆盖晚数、未覆盖晚数和权益未覆盖金额。普通订单完全隐藏“覆盖 0 晚”；会员模式确实无余额时显示“覆盖 0 晚、未覆盖 N 晚”，不能让人误解为本次住宿是 0 晚。
- 预订确认冻结；CHECK_IN 成功时将 HELD 转为 CONSUMED；入住前取消/未到释放；入住前缩短释放减少日期；已核销权益不被普通取消、缩短或退房隐式恢复。已入住后续住的新增覆盖在续住确认事务中立即核销。

**自动化门禁：**

- 身份证号唯一、会员搜索、Feishu record_id 多申请关联同一 `memberId` 及外部引用去重在领域、PostgreSQL 和 contract 测试通过。
- 三种会员产品的默认价、稳定 code、权益类型、精确房型映射和整房排除均有金标；不匹配房型失败关闭且零权益、零库存写入。
- 未收款不能生效；收款差额只提示；生效幂等重放只发放一个 30 夜 Lot；有效期精确从服务端生效时点起算一年。
- 多笔企微收款各自保存非空交易号，但不施加未经确认的全局唯一约束；收款更正、权益更正均保留原事实并追加冲销/调整，Query、Receipt 和审计链可完整追溯。
- 全额覆盖、部分覆盖、0 余额、续充后刷新、预订冻结、CHECK_IN 核销、取消/缩短释放和已核销不恢复均通过真实 PostgreSQL 验收。
- 同一住宿日期不得重复覆盖；并发会员订单或住宿订单不得重复生效、重复发放或超额冻结。
- 未勾选会员时不得读取、冻结或核销任何会员权益；API 可复算 coverageSet，Web 中文摘要与其精确一致。
- 会员住宿省略渠道可成功创建；强行提交任一渠道必须在预检、确认和数据库层失败且零业务写入；非会员普通住宿省略渠道继续失败。
- 桌面与手机 E2E 覆盖会员住宿表单、中文核对/成功、恢复语义和订单详情，确认无渠道字段、英文协议名、内部 ID 或横向溢出。
- 桌面与手机 E2E 选择会员后核对四项预填值，再逐项修改并断言 CREATE_ORDER 使用修改后的主要居住人快照。

**阶段 4 的局部业务 gate：**

同一会员若同时存在多个可覆盖同一日期、同一权益类型的有效 Lot，优先消耗哪一份额度会影响到期和余额事实。当前用户尚未确认分配顺序，不得自行选择“最早到期优先”、创建时间优先或任意数据库顺序。会员档案、会员订单、企微收款、生效和单一 Lot 旅程可先实施；开始多 Lot 自动 coverage 前必须确认唯一顺序并加入领域金标。

**用户人工验收：**

1. 打开会员列表，创建会员并按姓名、身份证号、手机号和微信号分别查找；重复身份证号必须明确拒绝。
2. 分别创建三种会员订单，核对默认价、30 夜权益和允许房型；改变成交价时确认必须填写原因。
3. 在一张订单登记两笔不同企微交易号的收款；合计与成交价不等时只看到差额提示，没有自动改价或自动结清。
4. 尝试无收款生效应被拒绝；录入至少一笔后点击“生效”，确认有效期从本次点击起算一年且只发放一次 30 夜。
5. 更正一笔交易号/金额和一次“更正后剩余间夜数”，确认旧记录、冲销/调整、新记录和当前余额都能查到。
6. 打开普通住宿，确认默认没有会员搜索且非会员模式要求渠道；勾选后选择会员，确认渠道字段消失，四项主要居住人资料自动预填且可以分别修改，再查看覆盖摘要。取消勾选后渠道恢复且不残留会员事实。
7. 使用 `D01`、B01、`101A` 验证三个会员产品；尝试不匹配房型和四人间整房组合时明确拒绝且余额不变。
8. 点击“核对并创建订单”，确认核对、成功页和订单详情没有渠道、英文协议名或内部 ID，订单详情显示“住宿来源：会员权益”；再入住并确认预订时冻结、入住时核销。

**停点：** 用户确认会员列表、会员订单、多笔企微收款、生效、余额及会员住宿通过后，才能进入阶段 5。

### 阶段 5：多人间首页显示全部住客昵称

**目标：** 多人间有多个床位被住宿订单占用时，父房逐日格直接显示每一个住客昵称，不再把第二个人以后压缩成“+N”。

**实施范围：**

- 复用服务端 `bedOccupancies[].occupants` 的逐床权威结果；先验证 DTO 完整性，现有 DTO 已满足时不做无意义的后端改造，缺字段时才进行 additive contracts/API/OpenAPI 扩展。
- 父房逐日格按 `inventoryUnitCode` 的稳定床位顺序显示每个有权查看的 `primaryOccupantLabel`。
- `1/4、2/4、3/4、4/4` 比例继续显示，所有昵称与比例必须同时可见；禁止“王羽 +1”“首个昵称 +N”、去重、隐藏后续昵称或仅在 Tooltip 中补全。
- 两个床位即使昵称相同也必须保留两项，不能因为文本相同而合并。历史缺失昵称按每个床位显示“历史未记录”，不得伪造昵称。
- 昵称可以在格内换行，父房行按内容统一增高；同一行所有日期列保持对齐，Hover、Tooltip 和状态变化不得再次改变尺寸或覆盖比例、日期及相邻格。
- Tooltip 和键盘详情继续显示床号、昵称、来源与日期，作为补充信息而不是完整昵称的唯一入口。
- 正常住宿和免费入住均计入昵称列表；维修、清洁和其他无居住人来源既不增加比例分子，也不得出现在住客昵称列表中。
- 桌面和平板房态格直接可见全部昵称；移动任务式界面的房间摘要或紧邻详情也必须直接列出全部有权昵称，不得恢复成 `+N`。
- 正常或免费住宿的日期格只显示昵称、占用比例和“已预订/在住”等业务状态，不显示“阻断”“N 个阻断”或冲突数量；Tooltip、title 和 accessible name 同样使用工作人员语言，不能把隐藏的技术词转移给屏幕阅读器。
- 房态日期格和跨日住宿横条不显示 `order_...` 订单号；可见文字、Tooltip、title、accessible name 和移动摘要均只保留昵称及必要业务状态。订单号只在阶段 7 的右侧订单上下文中显示，底层稳定订单引用继续用于点击定位和整段选择。

**自动化门禁：**

- Web state/component tests 覆盖 1、2、3、4 张床分别占用，并断言所有昵称按床号稳定顺序可见。
- E2E 明确断言父房格不存在 `+1`、`+2`、`+3` 折叠文案，且 `2/4、3/4、4/4` 与对应全部昵称同时出现。
- 同名住客、历史空昵称、正常与免费混合占用及维修排除均有回归测试。
- 鼠标悬停、键盘聚焦、屏读 accessible name 与可见昵称一致，获权 redaction 不泄露不可见住客。
- 正常已预订与在住日期格、Tooltip 和屏读名称均不存在“阻断”“阻断库存”“Claim”或冲突数量；底层 DTO 仍保留对应事实且可售判断不变。
- 日期格、跨日住宿横条、Tooltip、屏读名称和移动摘要均断言不包含 `order_` 订单号；隐藏显示不得删除 DTO 中的稳定订单引用。
- 1440、1024、768、375、320 CSS px 和 200% zoom 下无昵称遮挡、比例覆盖、日期错位或页面级横向溢出。
- 后续 90 天虚拟化必须保留本阶段的完整昵称与可变行高语义，不能为了性能重新折叠成 `+N`。

**用户人工验收：**

1. 打开一个四人间，在同一天安排两位不同昵称的住客。
2. 在首页父房格直接看到两个完整昵称和 `2/4`，确认不再出现“第一个名字 +1”。
3. 继续占用第三、第四张床，分别确认三个、四个昵称与 `3/4、4/4` 同时可见。
4. 检查跨多个日期时，每一天只显示当天实际占用床位对应的全部昵称。
5. 使用两个相同昵称占用不同床位，确认两项都保留；再检查历史未记录昵称的明确提示。
6. 加入维修来源，确认比例和住客昵称列表不会把它算作住宿人。
7. 悬停和键盘聚焦仍可查看床号与来源，但不使用悬停时也已经能看到全部昵称。
8. 检查单人间和多人间的住宿横条，确认昵称和必要状态可见但订单号不出现；点击后仍能定位正确订单。
8. 检查王羽的已预订日期格、悬停提示和屏读名称，确认不再出现“1 个阻断”“阻断库存”或 Claim 等技术词，同时该床位仍不可被重复预订。

**停点：** 用户确认多人间每格的全部昵称、比例和布局通过后，才能进入阶段 6。

### 阶段 6：维修锁房与内部占用决策（内部占用暂不实施）

**目标：** 保持已经确认的维修锁房能力；用户尚未决定免费入住与内部占用的精确分类条件，因此内部占用只记录候选边界，不在确认前实施，也不阻断后续核心订单阶段。

**实施范围：**

- “维修锁房”保持独立类型和原因，通过服务端高风险命令创建 typed Block/Claim，不创建 Order、Stay、pricing revision 或资金事实。
- 维修释放只释放精确匹配的完整 Block；历史和审计保留；多人间父房 `已占/总床数` 的分子不包含维修来源。
- “内部占用”候选模型仍是房源、日期和原因，不显示住客、渠道、会员或金额，并以独立 `INTERNAL_USE` Block 表达；但这不是已确认经营事实，禁止在本阶段写入运行 schema、枚举或 UI。

**自动化门禁：**

- MAINTENANCE 的 DTO、权限、幂等、冲突和释放行为保持有效，并与正常/免费住宿执行整房和子床双向互斥。
- 维修不产生入住人数、Order、Stay、会员或资金事实。
- 静态与 contract 检查证明未确认的 INTERNAL_USE 没有被偷偷加入运行合同。

**用户人工验收：**

1. 在空闲房源创建一次维修锁房，确认只要求房源、日期和原因，不要求入住人、渠道、会员或金额。
2. 在多人间床位设置维修，确认父房住客占用比例不增加。
3. 释放完整维修，确认房态恢复且历史仍可查。
4. 确认页面没有把免费入住改名或暗中归类为“内部占用”。

**停点：** 维修锁房验收通过即可进入阶段 7。内部占用保持 `deferred_decision`；用户以后核定边界时再作为独立增量插入，不允许因此修改已通过的免费入住语义。

### 阶段 7：连续订单区块选择与订单上下文

**目标：** 工作人员点击房态中的已占订单或住客昵称时，一次选中该订单同一 Stay 的完整连续住宿，并在右侧直接看到订单概述和当前合法操作，而不是单日选区、报价草稿、schema 500 或技术信息墙。

**实施范围：**

- 修复 amendment payload 查询 DTO 与 schema 不一致导致的订单详情 500。
- 选择身份以服务端返回的稳定 `orderId`、`stayId` 或 typed `sourceReference` 为准，不以昵称相同、颜色相同、日期视觉相邻或客户端拼接格子判定同一订单。
- 点击已占连续区块、区块内任一天或该区块的住客昵称，均选择权威同一 Stay 的完整连续住宿。连续续住后的原日期和新增日期属于同一个订单选择容器，不拆成“原订单”和“续住订单”；容器内部必须保留可见、可查询的原始预订、续住、缩短和换房 segment/amendment 边界，不能渲染成一段无法审计来源的连续色块。
- 换房不打断连续 Stay；同一次住宿发生无缝换房时，同时高亮分布在原房源和新房源行上的全部住宿分段。不同订单、有至少一天日期间隔的新住宿、相邻但不同的 Stay，以及昵称相同的其他住客绝不能合并。
- 多人间父房格内每个可见昵称都是独立订单入口：点击“王羽”只选择王羽对应的订单/Stay。点击只表达 `已占/总床数` 的父格背景时，如当天存在多个订单，显示可访问的住客/订单选择列表，不任意选择第一笔订单。
- 若完整 Stay 部分落在当前可见日期窗口之外，只高亮当前窗口内的交集；右侧仍显示完整入住、退房日期、总晚数和全部住宿分段，并提供定位其他分段的操作。
- 右侧上下文明确分为来源模式：空白区间继续显示房源、日期和“创建正常住宿、创建免费入住、维修锁房”；已占正常或免费住宿切换为订单模式，维修切换为 Block 模式。未确认的内部占用不得出现入口。订单模式不得出现报价、会员搜索、住宿草稿日期、“应用选区”“清除选区”或创建订单控件。
- 房态页面以主表为首要工作区，响应式决策按房态实际可用内容宽度而不是未经扣减的浏览器 viewport 作出。只有主表和上下文都达到已验证的最小宽度时才固定双栏；典型笔记本和其他空间不足的桌面使用紧凑导航/间距/工具栏与可打开、可关闭的订单上下文抽屉，不要求浏览器缩放，不把右侧上下文简单移到主表下方形成超长往返。
- 订单上下文抽屉必须具有明确的打开/关闭控件、可理解标题、焦点圈定、`Escape` 关闭和关闭后焦点恢复；打开、关闭及在空白选区/订单模式之间切换不得丢失当前日期窗口、筛选、展开、滚动位置、整段选择或草稿。短屏布局应让房态主表使用扣除真实顶部区域后的剩余高度，避免页面滚动与矩阵内滚动形成不可控的双重滚动。
- 订单上下文首先显示工作人员需要的“当前住宿安排”，再按时间顺序显示“变更记录”：订单号、主要居住人姓名与昵称、当前状态、正常/会员权益/免费入住标识、完整入住和退房日期、总晚数、当前房间/床位及全部住宿分段。每条变更记录展示变更前后日期/房源、原因、操作人员、时间、对应计价修订及相关收退款事实；点击记录只定位该次变更的日期段，不把它变成另一张订单。渠道、住宿金额、会员权益摘要和资金事实摘要仅按该订单真实适用内容只读展示。
- 订单号的工作人员可见位置收敛到右侧订单上下文和完整订单详情；房态日历横条不重复显示。该展示收敛不得改变稳定 `orderId`/`stayId` 关联、整段高亮、点击定位或返回恢复。
- 根据服务端当前状态、权限和 `allowedActions` 只显示合法动作，并使用工作人员语言：办理入住、办理退房、提前退房或缩短住宿、续住、换房、取消订单、标记未到、登记收款、登记退款。客户端不得仅凭状态名称自行推导或放宽动作。
- “打开完整订单详情”保留为次要入口；房态右侧已经能够完成日常判断和找到下一步，不强迫工作人员先跳到技术详情页。
- 技术 ID、raw JSON、Preview、Confirm、Receipt、policy code、Lot/Coverage ID 移出工作人员主区域；稳定引用、审计和持久结果继续保留在 API 与获权审计入口。
- 订单上下文不展示“是否阻断、阻断事实、精确冲突、Claim”等协议字段。已占订单只用状态和住宿日期表达不可重复出售；空白选区遇到真实占用或异常时，以“哪个房源、哪段日期、为什么不能操作”的业务句子说明。
- 订单上下文必须查询权威 Order/Stay DTO；浏览器不得把相邻逐日格自行拼成住宿区间。必要时只做 additive DTO/OpenAPI 扩展，不破坏既有外围智能体合同。
- 后续阶段完成任一订单动作后必须刷新房态投影；若该 Stay 仍有效则保留整段选择及合理焦点，若状态或日期使原选择失效则用中文说明变化并返回可定位位置，该行为由阶段 8 至 13 分别验收。
- 房态已占区块和订单列表都能进入同一份业务详情；从完整订单详情返回时恢复原日期、筛选、展开、滚动位置、已选 Stay 和焦点。
- 本阶段只验收合法动作的可见性、中文标签和正确路由；入住、退房、续住、缩短、换房、取消、未到及收退款的表单、事务结果和动作后整段选择回归，分别由阶段 8 至 13 实施，不在本阶段提前夹带。

**自动化门禁：**

- 含多个 amendment、revision、coverage 和资金事实的真实订单均可序列化并打开。
- 点击多日订单任一天或昵称，都按稳定引用选中并高亮完整连续 Stay；连续续住后的原区间与新增区间一起选中，同时保留可识别的 segment/amendment 分界，且点击某条变更记录只能定位该段，不能改变订单选择身份。
- 同一 Stay 的换房分段能跨房源行共同高亮；相邻独立订单、存在日期间隔的新住宿和相同昵称订单均不合并。
- 选择已占订单时不发起 Quote 请求，右侧不存在报价、会员搜索、草稿日期或创建订单控件；选择空白区间时仍保留正确的创建/Block 动作。
- 点击不含订单号的日历住宿横条仍按稳定引用打开正确订单；右侧显示订单号，日历横条、Tooltip 和 accessible name 不显示订单号。
- 正常已预订/在住订单上下文不存在“阻断、Claim、冲突数量”等机器文案；冲突防护仍由服务端 DTO、Preview 重验和 PostgreSQL 互斥证明未被删除。
- 多人间父格的各昵称分别解析到自身订单；只有比例或格子背景且存在多单时出现可键盘操作的明确选择列表。
- 完整 Stay 超出可见窗口时，高亮可见交集且上下文仍返回完整日期与所有分段，不由浏览器猜测窗口外事实。
- 无权字段正确 redaction；动作集合、禁用原因与服务端状态和权限返回完全一致，READ 主体看不到写操作。
- 使用预置事实模拟房态 projection 外部刷新后，有效选择保留、失效选择说明、从详情返回恢复和陈旧数据失败关闭均有 E2E；真正执行订单动作后的回归留在阶段 8 至 13。
- 鼠标、键盘、屏幕阅读器和触控可以选择同一订单区块；桌面、平板、移动、320 CSS px 和 200% zoom 无重叠或技术信息挤压。
- Playwright 几何与截图回归覆盖 `1280x720`、`1366x768` 和 `1440x800`：主表首屏至少可见 4 行，页面无横向溢出，资源列、日期头、筛选和日期控件可达；固定双栏不得把主表压到验收最小宽度以下。抽屉打开后内容不越界，键盘焦点受控，关闭后选区、滚动位置和触发元素焦点恢复。

**用户人工验收：**

1. 在房态点击 `101A` 上王羽连续住宿中的任一天或“王羽”昵称，确认整段连续日期一次高亮，而不是只选中当天。
2. 检查日历住宿横条只显示王羽昵称和必要状态、不显示订单号；右侧显示王羽这笔订单的订单号、基础信息、完整日期、总晚数、住宿分段和当前可执行操作，并确认不再显示报价、会员搜索、日期草稿、“应用选区”或“清除选区”。
3. 打开预置在真实 PostgreSQL 中、已经包含连续续住 amendment 的验收订单；分别点击原日期和新增日期，确认两者都选择同一个订单容器，但原始预订与续住区间仍有清楚边界；点击续住记录只定位新增日期段。
4. 打开预置的无缝换房验收订单，确认原房源和新房源上的同一 Stay 分段同时高亮，右侧按时间顺序列出全部分段。
5. 准备一笔相邻独立订单和一笔相同昵称订单，确认两者都不会与王羽的 Stay 合并。
6. 分别点击预置为不同状态的正常、会员和免费订单，确认适用摘要正确，且右侧只显示服务端为各自状态允许的动作入口。
7. 从右侧打开完整订单详情再返回房态，确认日期、筛选、展开、滚动位置、整段选择和焦点仍在原处。
8. 在 `1280x720`、`1366x768` 和 `1440x800` 验收房态：确认主表不再被固定右栏挤成狭小窗口、首屏至少可见 4 行；打开订单操作抽屉完成查看，再用关闭按钮和 `Escape` 分别关闭，确认均回到原选区、滚动位置和焦点。

**停点：** 用户确认连续订单区块、跨房分段、订单上下文、返回恢复和动作入口均可理解后，才能进入阶段 8。

### 阶段 8：入住与退房

**目标：** 普通、会员和免费住宿都能从同一订单详情完成履约，房态用一致的人类语言准确反映“已预订 → 在住 → 已退房”，并独立显示退房后清洁任务。

**实施范围：**

- 订单尚未入住时，房态显示“已预订”且订单上下文显示“办理入住”；`CHECK_IN` 成功后，活动 Stay 区间立即改为“在住”且上下文显示“办理退房”。
- 正常和免费订单按状态更新 Stay；退房产生后续清洁事实但不改写历史住宿分段。
- `CHECK_OUT` 成功后订单详情显示“已退房”，不再把订单表达为当前在住；退房日有清洁任务时房态独立显示“待清洁”，完成清洁后只更新清洁任务状态。待清洁是否阻断可售仍是局部业务 gate，不能在确认前把两者硬编码为同一状态机。
- 会员订单 CHECK_IN 时将现有 HELD coverage 转为 CONSUMED；CHECK_OUT 不重复核销。
- 页面用业务结果说明成功或失败，不展示 Confirm/Receipt 代码。

**自动化门禁：**

- `RESERVED -> CHECKED_IN -> CHECKED_OUT` 状态迁移、重复请求和非法状态零写入在 PostgreSQL 通过。
- 会员 CHECK_IN 只核销仍为 HELD 的 coverage，CHECK_OUT 和幂等重放不重复核销。
- 正常与免费住宿的房态 typed source、昵称和多人间占用比例在 E2E 中正确变化。
- E2E 断言工作人员可见住宿状态按“已预订 → 在住 → 已退房”变化，清洁任务独立显示“待清洁/已完成”，且不出现 `RESERVED`、`CHECKED_IN`、`IN_HOUSE`、`CHECKED_OUT` 或“阻断库存”等内部状态词。
- 订单详情的 allowed actions 与服务端状态机一致。

**用户人工验收：**

1. 分别对普通、会员和免费订单办理入住。
2. 确认房态由“已预订”立即变为“在住”，不出现另一套“已入住/入住中”同义状态，且住客昵称和床位占用比例正确。
3. 检查会员订单的权益在入住后已核销，普通和免费订单不触碰会员权益。
4. 办理退房，确认订单显示“已退房”、清洁任务显示“待清洁”；完成清洁后确认任务变为“已完成”，本阶段不预判它是否影响可售。
5. 入住或退房后返回房态，确认仍有效的 Stay 继续按完整区块选中；已结束而失效的选择得到明确说明和合理焦点。

**停点：** 用户确认入住/退房旅程通过后，才能进入阶段 9。

### 阶段 9：续住

**目标：** 工作人员能在订单原计价政策下延长连续住宿，并清楚看到变更前后日期和金额。

**实施范围：**

- 续住输入新的退房日期和本次原因，预检展示原日期、新日期、新增晚数、累计总晚数、原金额和重算后金额。
- 按原始入住日至新退房日的完整累计晚数重新选档，继续使用成交时锁定的 pricing policy version；跨月不拆分。
- 新 revision 不继承旧手工目标价。付费订单可在本次续住概述中重新勾选“调整最终总价”，同时展示政策基础价、目标价和差额，并单独填写调价原因；未勾选时使用政策重算金额。
- 已入住会员续住产生的新覆盖在续住确认事务中立即从 HELD 转为 CONSUMED，不等待第二次 CHECK_IN。
- 免费入住续住金额保持 0，并要求本次变更原因。

**自动化门禁：**

- 连续续住跨 7/14/30 夜档、跨自然月和允许跳档倒挂均使用原锁定政策版本。
- Preview 后库存、价格 basis 或权益变化时确认陈旧且零部分写入；幂等重放只返回原结果。
- 已入住会员的新增 coverage 在续住同一事务中立即 CONSUMED，不存在第二次 CHECK_IN 依赖。
- 免费续住追加 amendment/revision/audit，金额为 0 且无会员权益事实。

**用户人工验收：**

1. 对未入住订单续住并跨过 7/14/30 夜档，核对累计晚数和新金额。
2. 对已经入住的会员订单续住，确认新增覆盖立即核销。
3. 对免费入住续住，确认金额仍为免费且必须填写变更原因。
4. 对曾经调过价的付费订单续住，确认旧调价不会自动继承；本次只有重新勾选调价才出现新的目标价和调价原因。
5. 检查旧日期、旧金额和原始预订边界仍保留在变更历史中，新的计价修订可作为后续补收或退款事实的依据。
6. 续住成功后回到房态，点击原日期或新增日期，确认原区间与新增区间作为同一个连续 Stay 共同高亮，同时仍能看出并定位本次续住边界。

**停点：** 用户确认续住的累计计价和会员权益行为通过后，才能进入阶段 10。

### 阶段 10：缩短与人工退款提示

**目标：** 缩短后按新的完整住宿总晚数重选档位，并只把正差额提供给工作人员作为人工退款参考。

**实施范围：**

- 缩短输入新退房日期和本次原因，概述显示原日期、新日期、减少晚数、重算金额和 collectionDifference。
- 使用原锁定政策版本，按缩短后的完整连续住宿晚数重新选 1/7/14/30 夜档。
- 新 revision 不继承旧手工目标价；免费入住金额继续为 0。
- 付费订单可在本次缩短概述中重新勾选“调整最终总价”；只有勾选后才显示目标价、差额和必填调价原因。住宿变更原因仍独立必填，不能代替调价原因。
- 入住前会员缩短释放减少日期的 HELD；已经 CHECK_IN 核销的 CONSUMED 权益不得由普通缩短恢复。
- 只有 `netRecordedCollection - newCurrentContractAmount > 0` 时显示“可供人工退款处理的差额”；系统不自动退款、不连接支付平台。
- 真正退款仍需工作人员引用原 COLLECTION 并录入退款自身 transactionReference；系统还必须自动绑定本次缩短产生的计价修订，工作人员不选择或填写内部 revision ID。

**自动化门禁：**

- 缩短后的完整总晚数重新落档，使用原政策版本，并且新 revision 不继承旧手工调整。
- `netRecordedCollection` 高于、等于和低于新金额三种情况均保持 collectionDifference 原语义；只有正差额进入人工退款提示。
- 入住前缩短只释放减少日期的 HELD；已 CONSUMED coverage 不恢复、不改写。
- 免费缩短保持 0 元并追加完整历史；并发冲突和幂等重放无重复 revision。

**用户人工验收：**

1. 将一笔长住订单缩短到不同价格档位，核对新总晚数和新金额。
2. 分别测试净收款高于和低于新金额，确认不会把负差额显示成可退款金额。
3. 对入住前会员订单缩短，确认未入住日期的冻结权益释放。
4. 对已入住会员订单缩短，确认已核销权益不会自动恢复。
5. 对曾经调过价的付费订单缩短，确认旧调价不继承；只有本次重新勾选才要求新的调价原因。随后发起人工退款，确认它同时显示为本次缩短的退款并引用选定的原收款。

**停点：** 用户确认缩短重算、权益和退款提示通过后，才能进入阶段 11。

### 阶段 11：换房

**目标：** 换房保留同一个连续 Stay，并根据完整累计晚数对所有住宿分段重新计价。

**实施范围：**

- 工作人员选择换房生效日期、新房源和原因，概述显示原房源、新房源、分段日期和重算金额。
- 换房不打断连续住宿；以原始入住日至最终退房日的累计总晚数选择一次档位。
- 各住宿分段按该统一档位下对应价格产品精确计算，所有分段相加后只对最终总价 half-up 一次。
- 同价产品换房金额不变；跨产品换房生成新的完整 revision 和 Stay segment。
- 整房与子床目标继续按完整剩余区间事务校验；不得自动换房或静默缩短。
- 免费入住换房金额保持 0；会员 coverage 遵守已核销不可隐式恢复规则。
- 付费订单可在本次换房概述中重新勾选“调整最终总价”；只有勾选后才显示目标价、差额和必填调价原因，新 revision 不继承换房前的旧调价。
- 床位与整房之间的换房验收当前只适用于现金订单和免费入住。会员订单仅允许权益类型匹配的同 kind 换房；BED_NIGHT 与 ROOM_NIGHT 不得静默转换，跨 kind 会员换房在用户确认新规则前必须失败关闭且零业务写入。

**自动化门禁：**

- 同价产品换房、跨价格产品换房、现金/免费订单的床位到整房，以及跨档累计晚数均由领域金标覆盖。
- 会员跨 BED_NIGHT/ROOM_NIGHT kind 换房以稳定错误拒绝，原订单、coverage 和库存保持不变。
- 所有分段精确求和后只对完整住宿最终总价 half-up 一次，不逐段舍入。
- 目标房源整房/子床冲突在真实 PostgreSQL 中拒绝且零部分写入。
- MOVE 只追加 segment、amendment 和 revision；原分段、旧 revision 和已核销 coverage 不被覆盖。

**用户人工验收：**

1. 完成一次同价产品换房，确认金额不变且住宿连续。
2. 完成一次跨价格产品换房，核对分段摘要和最终一次舍入金额。
3. 使用现金或免费订单测试床位到整房及不同销售产品的换房冲突。
4. 使用会员订单尝试 BED_NIGHT 与 ROOM_NIGHT 跨 kind 换房，确认系统明确拒绝且不转换权益。
5. 对曾经调过价的付费订单换房，确认旧调价不继承；只有本次重新勾选才要求新的调价原因。
6. 检查原房源、原分段和旧 revision 没有被覆盖。
7. 换房成功后回到房态，点击任一分段，确认原房源与新房源上的同一 Stay 分段共同高亮，右侧仍显示一个订单上下文。

**停点：** 用户确认换房操作和分段计价通过后，才能进入阶段 12。

### 阶段 12：取消与未到

**目标：** 取消和未到清晰结束预订，并正确释放尚未履约的库存与会员冻结权益。

**实施范围：**

- 只在合法状态显示取消或未到，并要求本次原因。
- 入住前取消/未到释放全部 HELD coverage 和未来库存 Claim。
- 已经 CONSUMED 的权益不得由普通取消、未到或缩短恢复。
- 免费入住取消同样追加 amendment/revision/审计，金额保持 0。
- 页面明确显示“已释放什么、未改变什么”，不使用含糊的“操作成功”。
- 今日履约必须显示 `arrival < businessDate`、仍为 RESERVED 且尚未超过离店日的应到未到订单，不能等到退房日之后才出现在异常中；入口可以归入清晰的“异常/待处理”队列，但具体动作仍由服务端 `allowedActions` 决定。

**自动化门禁：**

- 取消与未到使用独立订单夹具，分别验证状态、库存释放、HELD 释放和 CONSUMED 不恢复。
- 非法状态、陈旧预检、重复请求和权限不足均失败关闭且零新增业务事实。
- 免费订单取消仍追加 amendment/revision/audit，金额保持 0 且不产生会员事实。
- `arrival = businessDate`、`arrival < businessDate <= departure`、`businessDate > departure` 三组预置订单在 TodayPage 与房态任务投影中的分类一致，不再存在只在一个视图可见的盲区。

**阶段 12 的局部业务 gate：**

用户尚未确认何时从“今日应到”转为“可标记未到”，以及营业日期与实际时钟的精确边界。可先修复订单可见性，但在该矩阵确认前不得自行开放或自动执行“标记未到”；绝不允许系统自动把逾期预订改成未到。

**用户人工验收：**

1. 使用第一笔未入住普通订单执行取消，确认房态恢复可售。
2. 使用另一笔独立的未入住会员订单标记未到，确认冻结权益全部释放且核销为 0。
3. 对不允许取消或未到的状态，确认页面不显示错误动作。
4. 检查历史原因和原订单仍可追溯。
5. 把一笔应到订单保留为未入住直到下一营业日，确认它仍出现在工作人员待处理队列，而不是从所有当天视图消失。

**停点：** 用户确认取消/未到的库存和权益释放通过后，才能进入阶段 13。

### 阶段 13：人工收退款、订单资金摘要与住宿收款转会员

**目标：** 资金页面只表达 PMS 人工记录的经营事实，不暗示外部到账、结清或自动核销；在住宿资金基础收口后，提供受限、可审计且原子化的住宿收款转会员旅程。

**实施范围：**

- 每笔 COLLECTION 和 REFUND 分别录入自身 transactionReference；退款继续引用原 COLLECTION。服务端还必须自动保存该笔资金事实所依据的不可变 pricing revision，不要求工作人员查看或选择 revision ID。
- 工作人员端所有住宿收款、退款和调价金额统一按人民币“元”输入和显示，再用十进制定点规则精确转换为 API/数据库的 minor units；不得要求工作人员为 `¥1,280` 输入 `128000`，也不得用二进制浮点乘法制造分值误差。
- 订单详情完整列出多笔收款、退款和冲销，不在 Order 上设置单一会被覆盖的交易号；同一计价修订可以关联多笔分次收款或退款。历史资金事实若没有计价修订关联，Query 原样返回 `null` 并由 Web 派生显示“历史未关联”，不得虚构回填。
- 主摘要使用“住宿金额、已记录净收款、差额”中文名称，并保留既有三金额事实语义。工作人员在原始预订、续住、缩短或换房记录下看到的是“与本次变更相关的收退款”，而不是内部 ID 或会计分摊结果。
- 成功结果不强制进入 Receipt/Command 技术页；网络中断时用“正在查询刚才的操作结果”恢复原命令。
- 普通住宿不新增未经确认的支付渠道；会员订单已确认的企微收款只作为该业务的稳定事实。两者都不建设微信接口、Webhook、自动到账、自动退款或银行对账。
- 普通住宿现有 `method` 的真实业务含义尚未确认。实施时不得把订单来源 `WECOM` 当作支付方式，也不得因为会员订单只收企微而把普通住宿强制改成企微；在用户确认前保持现有数据兼容并把该字段隔离为局部 gate。
- 在资金输入、交易号和 `method` 边界关闭后，增加受限业务操作“将住宿收款转入会员订单”：目标会员订单必须关联选定 `memberId`，住宿主要居住人的身份证号必须与会员唯一身份证号一致；只能选择匹配会员产品房型的一笔或多笔完整原 COLLECTION，不允许拆分单笔收款、复制原交易号或建设通用跨订单分摊。
- 原住宿收款事实保持不可变；转出/转入以新的双向可追溯事实引用原 COLLECTION 和目标 `membershipOrderId`。同一笔收款只能成功转入一次，转入后不得继续作为住宿退款来源或再次计入另一会员订单。
- 只有已经明确记录为企业微信且有真实 `transactionReference` 的住宿收款可进入候选列表；不得从订单来源渠道推断支付方式。差额收款作为会员订单的新企微 COLLECTION，必须录入自身交易单号。
- 工作人员显式确认后，在一个 PostgreSQL 事务中完成：资金转入、会员订单生效、30 夜 Lot 发放、本次住宿的专用 conversion consumption、剩余权益计算，以及住宿订单的新计价修订。任何一步失败时全部回滚。
- 专用 conversion consumption 从新发 Lot 立即核销本次实际住宿 N 夜并逐日关联该 Stay，作为“办会员时用新权益结算本次住宿”的审计事实；它不是普通 coverageSet，不宣称会员在过去日期已经生效，也不能用于其他历史住宿。剩余权益仍只在会员订单生效后的有效期内使用。
- 住宿新计价修订保留政策基础报价与原现金计价历史，记录本次会员转换后 `cashRemainder=0`、`currentContractAmount=0`；原收款历史仍可见，但经转入后不能在住宿与会员订单两边重复计入。存在退款、冲销、超出会员成交价或已转入状态的收款先失败关闭。

**自动化门禁：**

- 新 COLLECTION/REFUND 缺少自身 transactionReference 或无法绑定本次 Preview 所依据的 pricing revision 时，在领域、数据库和 API 均拒绝且零写入；确认前 revision 变化必须按陈旧 Preview 拒绝。
- REFUND 必须引用同订单原 COLLECTION，同时保存退款自身交易号和本次依据的 pricing revision；REVERSAL 保持现有无交易号语义并通过被冲销 Fact 间接追溯。
- transactionReference 不施加未经确认的全局唯一约束，不接受 Fact/Receipt/Command/correlation/idempotency 标识代替；pricing revision 关联只表示本笔人工资金事实的业务依据，不得解释成分摊、到账、结清或核销。
- 幂等重放不重复产生资金 Fact；Query/Receipt/OpenAPI 能从原始预订或具体变更记录追溯到最终资金事实，也能从退款追溯到原收款。
- 金额解析覆盖整数元、角、分、非法精度、大额和空值；Web 显示、API minor units、数据库事实和 Query 回显精确往返，收 `¥1,280` 不会成为 `¥12.80` 或 `¥128,000`。
- 转会员的身份证同一人、房型匹配、整笔金额相等、防重复转入、转入后不可再作住宿退款来源、差额新交易号和原子回滚均在真实 PostgreSQL 通过；Query/Receipt/审计可双向追溯住宿收款与会员订单。
- conversion consumption 只允许命令关联的本次实际住宿，拒绝其他已退房订单或任意历史日期；它不写入普通 coverageSet、不回溯会员生效日，剩余权益有效期仍从生效时点起算一年。
- 金标示例通过：公卫单人间住宿 7 夜、原企业微信收款 `¥590`；整笔转入默认成交价 `¥1,620` 的公卫单人间会员订单，再收差额 `¥1,030`；发放 30 ROOM_NIGHT，专用事实立即核销本次 7 夜，剩余 23 夜；住宿新 revision 的 cash remainder/current amount 为 0，原现金计价、原收款和原交易号历史仍保留且不重复计入。

**阶段 13 的局部业务 gate：**

- 会员身份证号是唯一业务键，但历史住宿的主要居住人快照可能没有身份证号。实现不得只凭姓名或昵称自动合并；身份证号能够精确匹配的旅程可先完成，缺失身份证号的转会员必须失败关闭，直到用户确认受控关联方式。
- 普通住宿资金 `method` 的真实含义必须先由用户确认，才能据此筛选“已经明确记录为企业微信”的可转入收款；不得把订单来源 WECOM 代替资金事实。

**用户人工验收：**

### 阶段 15：完整旅程回归与发布验收

**目标：** 证明所有分步改动可以组合成同一套稳定经营流程，而不是各阶段孤立演示。
